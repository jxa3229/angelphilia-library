# MJD 娃娃身体尺寸表与组合计算页（功能性新增）

## 状态

- 状态：待实施
- 类型：功能性新增
- 目标：先建立官方/可核验的身体配件尺寸底表，再新增用户可自行更换身体配件的组合计算页。

## 当前问题

目前仓库只保存了馆藏条目、详情页和索引页，没有面向 MJD 娃娃身体配件的结构化尺寸数据，也没有“按部位组合后计算最终身体参数”的页面。用户如果想确认某套配件组合最终会落到什么尺寸，只能手工对照零散资料，容易出现：

- 配件型号与尺寸分散在不同记录里，不易统一查询。
- 组合时无法明确每个最终尺寸来自哪个部位。
- 缺少“待核”机制，未确认的数据容易被误当成已知事实。

## 方案选项

### 方案 A：先做底表，再做计算页（推荐）

先抽出结构化基础数据，再在其上叠加组合计算页。

优点：
- 数据口径先稳定，后续计算逻辑更容易收敛。
- 缺失项可以直接进入待核清单，便于持续补全。
- 计算页可以完全依赖底表，不需要额外猜测。

缺点：
- 首版可见功能较少，用户要先等数据表成型。

### 方案 B：先做计算页，再反推数据表

先把交互页搭出来，再慢慢补底表。

优点：
- 页面体验更快可见。

缺点：
- 很容易出现“页面先行、数据不准”的情况。
- 规则会被 UI 形态牵着走，后续返工概率高。

### 方案 C：只做静态整理，不做计算页

只整理成可查表，不提供组合结果。

优点：
- 实现最简单。

缺点：
- 无法满足“自行更换配件后直接算结果”的目标。

## 方案设计

### 1. 数据底表

建议新增一份独立的结构化数据源，用于记录官方或可核验公开来源的配件尺寸。字段至少包括：

- `partType`：部位，如头、胸、腰、臀、大腿、小腿、上臂、下臂、手、脚
- `model`：型号或配件名称
- `bodySystem`：适用体型/系列
- `dimensions`：关键尺寸
- `skin`：肤色或版本
- `sourceUrl`：来源链接
- `note`：备注
- `confidence`：置信度或待核状态

底表只收录可核验来源，不把玩家主观估测写成正式数据。

### 2. 组合规则

组合计算只做确定性拼接，不做自动推算。每个最终输出字段都要明确来源：

- 选择了哪个部位配件
- 该字段从哪个部位继承
- 是否存在缺失或待核

当某个部位没有已知尺寸时，结果保留空位并给出缺失提示，不输出伪结果。

### 3. 组合计算页

新增一个独立页面，用于用户逐项选择配件并查看最终结果。页面建议分为三块：

- 左侧/上方：部位选择区
- 中间：最终身体参数区
- 下方：来源明细与待核提醒区

建议把当前选择写入 URL，便于刷新后恢复和分享。

![组合计算页信息结构](./images/01-body-calculator-structure.png)

### 4. 路由与入口

在现有静态站点路由中增加新页面入口，并在侧边导航中提供可发现的入口。页面标题和导航文案保持与现有站点风格一致，避免把新功能做成孤立工具页。

### 5. 维护方式

后续新增官方型号时，只追加底表数据，不修改计算逻辑。规则变化只在数据字段和映射关系层处理，避免页面与规则互相耦合。

## 实现要点

- 数据层优先：先把配件数据字段定死，再写计算逻辑。
- 来源优先：没有来源的字段不能进入正式底表。
- 结果透明：每个最终尺寸都必须可追溯到来源配件。
- 首版保守：不做跨部位插值、比例修正或自动补值。
- 预留扩展：后续可加入筛选、对比和收藏组合，但不作为首版范围。

## 验收标准

- [ ] 能列出一版结构化的身体配件底表。
- [ ] 底表中的记录都带有部位、型号、来源和尺寸字段。
- [ ] 组合页能按部位选择配件并输出最终身体参数。
- [ ] 每个输出字段都能追溯到具体来源配件。
- [ ] 缺失数据会明确提示“待核”，不会生成伪结果。
- [ ] `npm run build` 通过。

## 约束一致性

- 方案保持静态站点模式，不引入后端。
- 数据准备放在前端可维护层，符合仓库“构建前数据准备、运行时读取静态数据”的约束。
- 所有文件读写保持 UTF-8。
- 首版只收录官方或可核验公开来源，符合当前资料库的可信度边界。
